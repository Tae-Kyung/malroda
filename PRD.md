# [PRD] 말로다(MALRODA) 제품 요구사항 정의서

## 1. 제품 개요 (Product Overview)
**말로다(MALRODA)**는 디지털 소외 계층인 농어민이 일상 언어(음성)만으로 비즈니스 데이터를 쉽게 관리하고 자산 가치를 극대화할 수 있도록 돕는 플랫폼입니다.

### 1.1. 제품 비전 (Product Vision)
디지털 소외 계층인 농어민이 일상 언어(음성)만으로 비즈니스 데이터를 관리하고 자산 가치를 극대화하는 플랫폼.

### 1.2. MVP 주요 목표 (MVP Goals)
1. **베타 테스트 플랫폼 구축 (Web)**: 카카오톡 채널 연동 전, 실제 사용자들이 웹상에서 회원가입 후 챗봇과 대화하며 테스트할 수 있는 사용자 환경 및 관리자 대시보드 구축.
2. **음성 기반 재고 관리**: 정확하고 직관적인 재고 입출고 기록 및 조회 파이프라인(NL2SQL, Tool Calling) 구조 확립.
3. **사용성 분석 체계 확립**: 관리자가 사용자들의 "질문(프롬프트)"과 "AI의 의도 분석 결과 및 SQL 변환 내역"을 실시간으로 디버깅하고 모니터링할 수 있는 체계 구축.

## 2. 사용자 페르소나 및 다각화된 시나리오

### 2.1. 사용자 페르소나
- **농장주 (Primary)**: 현장 작업 중 손을 쓰기 어렵고 앱 조작이 번거로운 40~70대 농민.
- **농장 직원 및 가족 (Secondary)**: 사장님과 실시간으로 재고 현황을 공유하고 보고해야 하는 공동 작업자.
- **구매자 (B2B)**: 신뢰할 수 있는 지역/산지의 실시간 재고 정보를 찾는 유통업자.

### 2.2. 핵심 사용 시나리오 (User Scenarios)
농번기, 일상, 의사결정 등 농가 현장에서 발생하는 특수한 맥락을 지원하는 6가지 세부 시나리오입니다.

#### 1) 일상적인 입출고 및 수정 (가장 빈번한 케이스)
- **입고**: "오늘 수확한 부사 사과 50박스 1동 창고에 넣었어."
- **출고**: "대관령에서 로즈마리 3kg 식당으로 나갔으니까 재고에서 빼줘."
- **폐기**: "자스민 구역 카네이션 10단이 시들어서 아까 버렸어. 폐기 처리해."
- **정정**: "아까 서울 사과 20박스라고 했는데, 다시 세보니까 25박스네. 25박스로 고쳐줘."
- **단위 혼용**: "장미 5그루 심고, 백합 10단은 저온 저장고로 옮겼어."
- **모호한 표현**: "창고에 있는 거 절반 정도 나갔어." (이 경우 AI가 "정확히 어느 구역의 무슨 품목이 몇 박스 나갔나요?"라고 되물어야 함)

#### 2) 재고 확인 및 통계 (의사결정용)
- **현재 잔량**: "지금 우리 창고에 로즈마리 총 얼마 있어?"
- **품목별 조회**: "오늘 나간 거 말고, 남은 장미가 몇 송이지?"
- **기간별 통계**: "어제 하루 동안 출고된 사과가 총 몇 박스야?"
- **비교**: "저번 달에 비해서 이번 달에 폐기량이 얼마나 늘었어?"
- **부족 알림**: "재고 10개 밑으로 떨어진 품목들만 쭉 말해줘."

#### 3) 직거래 및 구매자 매칭 (수익 창출용)
- **구매자 탐색**: "지금 내 로즈마리 사겠다는 사람 근처에 있어?"
- **재고 공개**: "오늘 수확한 거 신선하니까 구매자들이 볼 수 있게 공개해줘."
- **매칭 확인**: "내 사과 재고 보고 연락 온 도매업자 있어?"
- **시세 문의**: "지금 서울 가락시장에서 장미 한 단에 얼마에 거래돼?"
- **판매 요청**: "재고 남은 거 빨리 팔고 싶은데, 직거래 장터에 올려줘."

#### 4) 조달청(나라장터) 정보 연동 (정보 격차 해소)
- **공고 검색**: "이번 주에 조달청에 우리 꽃 관련 입찰 뜬 거 있나 확인해줘."
- **상세 정보**: "아까 알림 온 학교 급식 사과 납품 공고, 마감이 언제까지야?"
- **입찰 추천**: "지금 내 재고 상황으로 참여해 볼 만한 조달 공고 추천해줘."
- **과거 낙찰 정보**: "작년 이맘때 군부대 납품 낙찰가가 얼마였는지 알아?"

#### 5) 시스템 설정 및 오류/예외 상황 (사용성)
- **작업 취소**: "방금 내가 말한 거 기록하지 마, 잘못 말했어."
- **공유 관리**: "우리 아들도 이 재고 같이 볼 수 있게 추가해줘."
- **위치 기반**: "지금 내가 있는 제2창고 재고만 보여줘."
- **알림 설정**: "재고 나갈 때마다 우리 와이프한테도 카톡 가게 해줘."
- **음성 인식 오류**: "아까 말한 게 제대로 안 들어간 것 같은데, 마지막 기록 다시 읽어줘."

#### 6) 농번기 특화 시나리오 (현장 밀착형)
- **일괄 처리**: "사과 다 나갔어. 재고 0으로 만들어."
- **단축 명령**: "장미 입고, 평소대로 100개." (기존 데이터를 기반으로 한 단축 명령)
- **지연 기록 (Batch)**: "바쁘니까 이따가 밤에 한꺼번에 물어볼게. 일단 지금은 로즈마리 5개 출고만 기록해."

## 3. 핵심 기술 요소 및 AI 엔진 튜닝 요구사항 (Technical Requirements)

### 3.1. 하이브리드 AI 아키텍처 (Hybrid AI Architecture)
안전성과 유연성을 동시에 확보하기 위해 라우팅 기반의 하이브리드 엔진 아키텍처를 도입합니다.
- **Frontend**: KakaoTalk Channel UI (챗봇 인터페이스) + Next.js (인앱 Webview)
- **Backend**: Vercel Serverless Functions 기반 하이브리드 API 분기 처리
- **Database**: Supabase (PostgreSQL) - RLS 뷰(View) 및 RPC (저장 프로시저) 적극 활용
- **AI Engine (OpenAI)**: 음성텍스트 변환(Whisper) + 의도 파악/로직 분기(GPT-4o-mini) + 동적 쿼리 생성(GPT-4o)

### 3.2. 💡 지능형 라우터 프롬프트 (Intelligent Router Prompt)
사용자의 입력(STT 결과)을 가장 먼저 받아 **5가지 핵심 카테고리 중 하나로 분류(교통정리)**하는 역할. 빠르고 저렴한 모델(GPT-4o-mini 등) 사용 권장.
1. `INVENTORY_UPDATE` (재고 변동): 입고, 출고, 폐기, 정정 명령
2. `INVENTORY_QUERY` (재고 조회): 잔량, 과거 기록, 통계 질문
3. `PROCUREMENT_INFO` (조달/입찰): 입찰 공고, 조달청 연동 질문
4. `BUYER_MATCHING` (구매/판로): 구매자, 매칭, 시세 질문
5. `GENERAL_SUPPORT` (기타): 인사, 사용법 문의, 잡담 처리

### 3.3. 💡 하이브리드 NL2SQL 엔진 설계 (읽기 vs 쓰기 분리)
라우팅 결과에 따라 데이터 무결성을 보장하기 위해 처리 방식을 이원화합니다.
- **쓰기 (Update/Write) ➡️ Tool Calling (Function Call) 방식** 
  - `INVENTORY_UPDATE` 액션에서 적용.
  - AI가 SQL을 짜지 않고, 추출한 JSON 데이터(품목, 수량, 단위)를 통해 정형화된 **Supabase RPC (저장 프로시저)**만 안전하게 호출. (예: `malroda_update_inventory(item, qty, type)`)
- **읽기 (Read/Query) ➡️ NL2SQL (Dynamic Schema Query) 방식**
  - `INVENTORY_QUERY` 액션에서 적용.
  - 다양한 통계 및 현황 질의에 대응하기 위해 AI가 동적 쿼리를 생성.
  - 🚨 보안 필수: 원본 테이블 접근 권한은 제한하고, AI 전용 **읽기 전용 View(예: `v_malroda_inventory_summary`)**에만 접근시켜 SQL 인젝션 및 데이터 훼손(DROP/DELETE)을 원천 차단.

### 3.4. 추가적인 튜닝 과제
1. **동의어 처리 (Synonyms)** & **단위 처리 (Unit Converter)**: 다양한 표현 및 단위 규격 정규화 보장.
2. **맥락 유지 (Session Memory)**: 이전 대화를 기억하기 위한 세션 스토리지 활용.

### 3.5. 개발용 웹 시뮬레이터 (Developer & QA Simulator)
### 3.5. 베타 테스트용 웹 플랫폼 및 관리자 대시보드
- 카카오톡 채널 정식 런칭 전에, 다양한 환경의 베타 테스터들이 직접 사용해볼 수 있는 Web App을 제공합니다.
- **[사용자 환경 (User Dashboard)]**: 자체 회원가입/로그인(Supabase Auth) 기능을 제공하며, 로그인 시 초기 농장(Farm) 설정과 재고 관리가 가능한 웹 기반 시뮬레이터 지원.
- **[관리자 대시보드 (Admin Dashboard)]**: 테스터들의 가입 현황 리스트, 시스템 전체 사용량 통계, 그리고 가장 중요한 **"채팅 로그(Raw Prompts & AI Responses)"**를 실시간으로 모니터링하고 분석하여 NLP 엔진을 고도화할 수 있는 관리자 전용 UI 모듈.

## 4. 데이터베이스 및 API 명세서

### 4.1. 데이터베이스 스키마 설계 (Supabase PostgreSQL)

#### 핵심 테이블
- **`malroda_profiles` / `malroda_farms` / `malroda_farm_members`**: 사용자 및 농가, 권한 정보 분리 관리
- **`malroda_items`**: 품목 명세 (품목명, 등급, **재배 구역/Zone** 단위 변환 기준 포함)
- **`malroda_inventory_logs`**: 입출고 히스토리
- **`malroda_session_contexts`**: 진행 중인 대화 맥락 보관 (단기 기억)
- **`malroda_prompt_test_cases`**: 프롬프트 검증용 데이터 (TC 데이터셋)

#### 하이브리드 엔진용 데이터 객체 (보안/효율)
- **[신규] RPC (저장 프로시저)**: `malroda_update_inventory()` 함수 생성. (서버/AI의 Tool parameter에 의해 호출되며 트랜잭션 단위로 안전하게 재고 등락 및 로그 기입을 수행)
- **[신규] Read-Only View (읽기 뷰)**: `v_malroda_inventory_summary` (농민 당 품목별 총 잔량, 최근 출고일 등을 조인해 둔 뷰. NL2SQL이 이 뷰만 참고하도록 하여 복잡한 쿼리를 간소화 및 Row-Level Security 보장)

### 4.2. 주요 API 명세서 요약 (Endpoint Specs)
| Method | Endpoint | 설명 | Request Body/Query | Response (Success) |
| :--- | :--- | :--- | :--- | :--- |
| POST | `/api/webhook/kakao` | 카카오톡 채널 수신 웹훅 | `User/Action Payload` | `200 OK` (카카오 응답) |
| POST | `/api/transcribe` | 음성 파일을 텍스트 변환 | `{ audio_url: "..." }` | `{ text: "...", sess_id: "..." }` |
| POST | `/api/router`| (신규) 텍스트를 5개 의도로 분류| `{ text: "..." }` | `{ intent: "INVENTORY_UPDATE", entities: {...} }` |
| POST | `/api/execute-update` | 의도=Update 시 RPC 호출 | `{ action: "IN", item: "사과", qty: 10 }` | 결과 및 확인 응답 텍스트 |
| POST | `/api/execute-query` | 의도=Query 시 NL2SQL 실행| `{ query_text: "어제 나간건?", ... }`  | Supabase DB조회 결과 텍스트 반환 |

## 5. 성공 지표 및 로드맵
위의 전략을 바탕으로 프롬프트 데이터셋을 구축하여 전체 시나리오 응답 성공률 95% 이상을 달성하는 것을 목표로 합니다.
- **Phase 1 (MVP)**: 웹 기반 베타 테스트 플랫폼 구현, 유저/어드민 대시보드, 하이브리드 엔진 최적화.
- **Phase 2**: 조달청 및 도매 시세 외부 데이터 연동, 카카오봇 채널 정식 출시.
- 이후 로드맵으로 어업 확장 및 B2B 직거래 마켓플레이스로 발전시킵니다.
